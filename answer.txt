1. Explain the overall design of your program and justify why you chose this particular class structure and set of user-defined methods.

The program is a simple console-based Monthly Budget Tracker.
I created one main class called BudgetCategory. Each object represents one budget category (example: "Food", "Transportation").

Attributes:
- name (string)
- limit (float) – monthly budget ceiling
- expenses (float) – total spent so far
- transactions (list) – keeps record of each expense (description + amount)

User-defined methods (minimum 2 required):
• add_expense(amount, description)  → adds spending and records the transaction
• status()  → returns colored/warning-style message about how much is left or if over budget

I also added helper method:
• remaining()  → computes limit - expenses

Justification:
- One class per category follows Single Responsibility Principle
- Easy to understand: real life = "each category manages its own money"
- Methods are meaningful: users add expenses → immediately see warnings
- Keeps code simple but still shows encapsulation, instance variables, and behavior inside objects

2. Describe how user input is handled in your program and discuss the role of exception handling (try–except) in preventing incorrect or unexpected behavior.

User input is collected using input() inside a while True menu loop.

Main places where input is taken:
- Menu choice (1,2,3,4)
- Category name
- Budget limit amount
- Expense amount
- Expense description (optional)

Exception handling used:
• try: float(limit_input)  → catches letters/symbols instead of numbers
• try-except when creating BudgetCategory (checks name not empty, limit > 0)
• try-except when adding expense (amount must be positive number)
• General except Exception as safety net
• Special except KeyboardInterrupt → friendly exit when user presses Ctrl+C

Role:
- Prevents program from crashing when user types wrong format (example: "five thousand" instead of 5000)
- Shows helpful messages instead of red traceback
- Makes program much more user-friendly and robust

3. Identify one limitation of your current implementation and explain how it could be improved using additional OOP concepts or better error handling.

Limitation:
All data is stored only in memory → everything disappears when the program is closed.
No way to save budgets or continue next time.

Possible improvements:
a) Create a new class BudgetManager that:
   - contains dictionary or list of BudgetCategory objects
   - has methods: save_to_json(), load_from_json()
   → uses json module to write/read file

b) Add more methods to BudgetCategory:
   - get_transaction_history()
   - reset_expenses()
   - increase_limit(new_amount)

c) Better input validation:
   - Use custom exception classes (class InvalidBudgetError(ValueError): ...)
   - Add maximum name length check
   - Prevent duplicate category names (already partially done with lower-case key)

d) Future OOP idea:
   Create abstract class BudgetBase → two child classes:
   • StrictBudgetCategory (cannot go over limit)

These changes would make the program more complete while still using proper OOP thinking.
